# チーム運用ルール

> 構造化作業をチーム（複数のClaudeエージェント / Codex / 人間）で実施する際のルールを定義する。

---

## 1. 役割定義

### 1.1 チーム構成

| 役割 | 担当 | 責務 |
|------|------|------|
| **オーナー** | 人間 | 設計書の承認、方針決定、最終品質判断 |
| **リード** | Claude Code（メインセッション） | 設計書作成、作業割当、レビュー統括、manifest管理、コミット |
| **ワーカー** | Claude Code（サブエージェント） | セクション単位の構造化作業（Step 1〜4） |
| **レビュアー** | Claude Code（別エージェント）/ Codex | Gate 3（粒度・構造）、Gate 4（原本忠実性）のレビュー |

### 1.2 権限と制約

| 役割 | できること | できないこと |
|------|----------|------------|
| ワーカー | YAML作成・編集、バリデーション実行 | 設計書の変更、manifest の status 更新、コミット |
| レビュアー | レビュー結果の報告、修正提案 | YAML の直接編集 |
| リード | 全操作 | 設計書の承認（オーナーのみ） |

## 2. 並列作業のルール

### 2.1 作業単位

- **基本単位**: 1セクション = 1ワーカーの1タスク
- **並列可能条件**: 異なるセクションは並列作業可能（ただし §2.3 の制約に従う）
- **並列不可**: 同一セクションの Step を複数ワーカーで分担しない

### 2.2 ID範囲の事前割当

並列作業時にIDが衝突しないよう、**作業開始前に**リードがID範囲を割り当てる。

```yaml
# 割当例（核心の場合）
worker_a:
  sections: [Kaku_01_01, Kaku_01_02, Kaku_01_03]
  node_id_range: khn-001 〜 khn-050
  sentence_id_range: kaku-01-* 〜 kaku-03-*

worker_b:
  sections: [Kaku_02_01, Kaku_02_02, Kaku_02_03]
  node_id_range: kbk-001 〜 kbk-050
  sentence_id_range: kaku-04-* 〜 kaku-06-*
```

**重要**: カテゴリが異なる章を異なるワーカーに割り当てれば、ID接頭辞が異なるため衝突リスクが低い。同一カテゴリの章を複数ワーカーに割り当てる場合は、番号範囲を明示的に分ける。

### 2.3 依存関係による作業順序の制約

- あるセクションBの `section_prerequisites` にセクションAが含まれる場合、**Aの構造化がGate 2まで完了してから**Bに着手する
- ただし、ノードの `prerequisites` は後から設定してよい（後述 §2.4）

### 2.4 後回しにしてよい作業

以下は並列作業の効率のために後回しにしてよいが、**必ず後から実施する**:

| 作業 | 後回しの理由 | 完了期限 |
|------|------------|---------|
| 章をまたぐ node_prerequisites | 他章のノードIDが未確定 | Phase 3（全体整合性チェック）まで |
| cross_book_links | 別作業として実施 | 構造化完了後の別フェーズ |

## 3. 情報共有プロトコル

### 3.1 ワーカー → リードへの報告

各セクション完了時に、ワーカーは以下の情報をリードに報告する:

```
## セクション完了報告: 〈セクションID〉

### 作成物
- knowledge_nodes: N件（ID: prefix-XXX 〜 prefix-YYY）
- sentences: M件
- understanding_goals: L件
- check_points: K件

### 品質ゲート結果
- Gate 1（情報完全性）: ✅ / ❌（詳細）
- Gate 2（自動バリデーション）: ✅ エラー0件 / ❌（エラー内容）

### 判断に迷った点
- （粒度、ノード分割、タグ付け等で迷った判断を記録）

### 次セクションへの申し送り
- （次セクションの作業者が知っておくべき情報）
```

### 3.2 リード → レビュアーへの依頼

Gate 3/4 のレビュー依頼時に、以下の情報を渡す:

```
## レビュー依頼: 〈セクションID〉

### 渡す資料
1. 構造化データ（YAML）
2. 参考書の書き起こしMD
3. 参考書別設計書（ノード抽出ルール・粒度基準の参照用）
4. Step 1 で作成した情報棚卸しリスト

### レビュー観点
- Gate 3 の場合: 粒度・依存関係・表記一貫性
- Gate 4 の場合: 情報ロス・情報歪み・タグ妥当性

### 期待する出力形式
- 問題なし / 要修正の一覧（修正箇所・理由・修正案）
```

### 3.3 レビュアー → リードへの報告

```
## レビュー結果: 〈セクションID〉 Gate 〈番号〉

### 総合判定: ✅ 合格 / ❌ 要修正

### 指摘事項（要修正の場合）
1. 【重要度: 高/中/低】〈指摘内容〉
   - 箇所: 〈ノードID or 英文ID〉
   - 理由: 〈なぜ問題か〉
   - 修正案: 〈具体的な修正内容〉

2. ...

### 良かった点
- （品質が高い部分のフィードバック — 他ワーカーの参考になる）
```

### 3.4 判断基準の共有（キャリブレーション）

並列作業の品質を揃えるため、以下の仕組みを使う:

#### パイロットセクションの共有レビュー

- パイロット（最初の2〜3セクション）の完成データを**全ワーカーに共有**する
- ワーカーは自分の作業開始前にパイロットデータを読み、粒度・記述スタイルを把握する

#### 判断事例の蓄積

構造化作業中に「迷った判断」が出たら、以下の形式で記録し、全チームに共有する:

```yaml
# decisions.yaml — 判断事例の蓄積
decisions:
  - id: 1
    date: "2026-02-28"
    section: "Kaku_01_01"
    question: "「核心」ステートメントをノードにするか、notes に入れるか？"
    decision: "ノード化しない。notes に「核心: 〈内容〉」と記述。理由: ステートメント単独では確認テストの問題を作れない。"
    decided_by: "リード"

  - id: 2
    date: "2026-02-28"
    section: "Kaku_01_02"
    question: "解説中の断片的英文（主語のみ、句のみ）を sentences に登録するか？"
    decision: "登録しない。完全な文として成立するもののみ登録。理由: テスト出題に使えない断片は対象外。"
    decided_by: "リード"
```

## 4. 作業ファイルの構成

### 4.1 ディレクトリ構造

```
scripts/supabase/
├── 〈参考書略称〉_data/          # 構造化データ格納ディレクトリ
│   ├── manifest.yaml            # 進捗管理
│   ├── decisions.yaml           # 判断事例の蓄積
│   │
│   ├── knowledge_nodes/         # 知識ノード（章ごと or セクションごと）
│   │   ├── ch01.yaml
│   │   ├── ch02.yaml
│   │   └── ...
│   │
│   ├── sentences/               # 英文（セクションごと）
│   │   ├── sec_01_01.yaml
│   │   ├── sec_01_02.yaml
│   │   └── ...
│   │
│   └── sections.yaml            # セクション一覧・マッピング
│
├── import_〈参考書略称〉.py      # DB投入スクリプト
└── validate.py                  # 自動バリデーション
```

### 4.2 ファイル分割の指針

| データ種別 | 分割単位 | 理由 |
|-----------|---------|------|
| knowledge_nodes | 章ごと | レビュー単位として適切。1ファイルが長大にならない |
| sentences | セクションごと | 1セクション完結の原則に合致 |
| sections | 参考書ごとに1ファイル | セクション間の依存関係を一覧で把握 |
| manifest | 参考書ごとに1ファイル | 進捗管理の単位 |

### 4.3 ファイルのロック（並列作業時）

- **同一ファイルを複数ワーカーが同時に編集しない**
- ワーカーは自分に割り当てられたセクション/章のファイルのみ編集する
- `sections.yaml` と `manifest.yaml` は**リードのみ**が編集する

## 5. Git ワークフロー

### 5.1 ブランチ戦略

```
main
  └── structuring/〈参考書略称〉        # 参考書ごとの構造化ブランチ
        ├── セクション単位でコミット
        └── 全セクション完了後に main へマージ
```

### 5.2 コミットルール

- **コミットタイミング**: 各セクションの Step 7（全品質ゲート通過後）
- **コミット単位**: 1セクション = 1コミット（原則）
- **コミットメッセージ**: `feat(構造化): 〈参考書略称〉 〈セクションID〉 知識ノードN件・英文M件`
- **コミット実行者**: リードのみ（ワーカーはコミットしない）

### 5.3 マージ条件

以下をすべて満たした場合にmainにマージする:
- [ ] 全セクションが completed 状態
- [ ] 全セクションが Gate 1〜5 を通過
- [ ] Phase 3（全体整合性チェック）が完了
- [ ] オーナー（人間）の承認

## 6. 進捗報告

### 6.1 定期報告（章の完了時）

```
## 進捗報告: 〈参考書名〉 Chapter N 完了

### 進捗サマリ
- 完了セクション: X / Y（全体の Z%）
- 知識ノード: N件（累計）
- 英文: M件（累計）

### 品質指標
- ノード/セクション比率: X.X（目安: Y.Y〜Z.Z）
- check_points/ノード平均: X.X
- Gate 4 での要修正率: X%

### 判断事例の追加
- N件の新しい判断事例を decisions.yaml に追加

### 次章の予定
- セクション数: N
- 割当: ワーカーA → sec_XX〜YY、ワーカーB → sec_ZZ〜WW
```

### 6.2 異常時の即時報告

以下の場合は章の完了を待たず、即座にリード（→オーナー）に報告する:

- 設計書の想定と大きく異なる構造パターンを発見した
- 粒度基準の目安範囲を大きく逸脱するセクションがある
- 書き起こしMDとPDFの間に重大な差異（ページ欠落等）を発見した
- Gate 3/4 で大量の要修正指摘があり、設計書の修正が必要と判断した

## 7. 品質保証の責任分担

| 品質ゲート | 実施者 | 判定者 |
|-----------|--------|--------|
| Gate 1（情報完全性） | ワーカー | ワーカー（セルフチェック） |
| Gate 2（自動バリデーション） | ワーカー | スクリプト（自動） |
| Gate 3（粒度・構造） | レビュアー | リード（レビュー結果を確認） |
| Gate 4（原本忠実性） | レビュアー | リード（レビュー結果を確認） |
| Gate 5（定量チェック） | リード | リード |

**不合格時のフロー**:
1. レビュアーが修正点を具体的に指摘
2. ワーカー（または別のワーカー）が修正
3. 修正後、不合格だったゲートを再実行
4. 合格するまで繰り返す

## 8. トラブルシューティング

### よくある問題と対処法

| 問題 | 対処法 |
|------|--------|
| ワーカー間で粒度が揃わない | パイロットデータを再共有し、具体例で基準を合わせる |
| 設計書に想定されていない構造パターン | 作業を止め、リードに報告。設計書を改訂してから再開 |
| Gate 4 で大量の情報ロスが指摘される | Gate 1 のセルフチェックが不十分。棚卸しリストの粒度を上げる |
| ID範囲が足りなくなった | リードが追加範囲を割り当て。既存IDは変更しない |
| 参考書内参照の先が未構造化セクション | notes に「参考書内参照: →p.XX」と記録し、後から prerequisites を設定 |
