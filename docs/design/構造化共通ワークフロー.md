# 構造化共通ワークフロー

> 全参考書に共通する構造化の手順・品質基準・情報保存ルールを定義する。
> 参考書固有の設計判断は「参考書別設計書」に記載する。
> チームでの並列作業ルールは「チーム運用ルール」に記載する。

---

## 0. ドキュメント階層

```
構造化共通ワークフロー（本書）
  ├── 全参考書で共通の手順・基準
  ├── 情報保存ルール
  ├── 品質ゲート定義
  └── データフォーマット仕様（YAML）

参考書別設計書（参考書ごとに1つ作成）
  ├── その参考書固有の構造分析
  ├── ノード粒度・カテゴリ体系・ID体系
  ├── 英文抽出ルール
  └── 特殊要素の扱い（図解・マーカー等）

チーム運用ルール
  ├── 並列作業のルール
  ├── レビュー手順
  └── エージェント間の情報共有
```

---

## 1. 原則

- **1セクション完結**: 1セクションの全工程（Step 1〜7）を完了してから次に進む。途中で飛ばさない
- **情報ロスゼロ**: 参考書に書かれている知識・情報を構造化の過程で失わない（→ §2 情報保存ルール）
- **参考書の体系を尊重**: 著者の視点・教え方の体系をそのまま構造化する。他の参考書に合わせて歪めない
- **設計先行**: 構造化作業に着手する前に、必ず「参考書別設計書」を作成し、人間のレビューを通す
- **セクション単位レビュー**: 品質を最優先し、1セクションごとに全品質ゲートを通す
- **PDF原本照合**: 書き起こしMDだけでなく、必ずPDF画像も確認する

## 2. 情報保存ルール

### 2.1 保存すべき情報の分類

構造化の過程で、以下の情報を**必ず**なんらかの形で保存する。

| 情報の種類 | 保存先 | 保存方法 |
|-----------|--------|---------|
| **概念の定義・説明** | knowledge_nodes の notes | ノードの説明として記述 |
| **理解目標** | understanding_goals | 「〜できる」形式で列挙 |
| **確認ポイント** | check_points | question/answer 形式 |
| **英文と和訳** | sentences の english/japanese | 正確に転記 |
| **構文分析** | sentence_structures | overall/details で記述 |
| **著者の教え方の順序** | sections の並び順 | セクション順で暗黙的に保存（OK） |
| **図解・樹形図** | 後述 §2.2 | notes に記述を残す |
| **難易度・頻出度マーカー** | 後述 §2.3 | メタデータとして保存 |
| **書内の相互参照** | 後述 §2.4 | notes または prerequisites |
| **補足・発展・例外** | 後述 §2.5 | notes または専用フィールド |

### 2.2 図解・樹形図の保存

図そのものをデータとして保存する必要はない。ただし、**その図が存在すること・何を表現しているか**を必ず残す。

**保存先**: knowledge_nodes の `notes`、または sentences の `notes`

**記述フォーマット**:
```
[図表: 〈図の種類〉〈図が表現している内容の要約〉（参考書 p.XX）]
```

**例**:
```yaml
notes: |
  [図表: 樹形図 — 文型判別の手順。自動詞→SV/SVC、他動詞→SVO/SVOO/SVOC の分岐を示す（p.45）]
  [図表: 等式図 — S=Cの関係を視覚的に示す。「主語の性質・状態を表す」（p.46）]
```

### 2.3 難易度・頻出度マーカーの保存

参考書に記載されている難易度・頻出度の情報は、**そのまま**保存する。

**保存先**: 参考書別設計書で定義するフィールド（例: sentences の `difficulty`、knowledge_nodes の `frequency`）

マーカーの種類は参考書ごとに異なるため、参考書別設計書で具体的なフィールドとマッピングを定義する。

**例（スクランブルの場合）**:
```yaml
# 参考書別設計書で定義
difficulty_markers:
  "基本": "basic"
  "標準": "standard"
  "発展": "advanced"
  "頻出": true  # frequency フラグ
```

### 2.4 書内の相互参照の保存

参考書内で「→p.XX参照」「→Theme XX」のような相互参照がある場合、必ず保存する。

**保存先**: knowledge_nodes の `notes` または `internal_references`（参考書別設計書で定義）

**記述フォーマット**:
```yaml
notes: |
  参考書内参照: →p.174「後置修飾は英語の大きな特徴」（テーマ19）
  参考書内参照: →p.300「関係詞の全体像」
```

**さらに、参照先が構造化済みのセクションの場合**:
- prerequisites や related として明示的にリンクを設定する

### 2.5 補足・発展・例外の保存

著者が「補足」「プラスα」「注意」「例外」として記述している情報は、以下のいずれかで保存する:

| 情報の性質 | 保存方法 |
|-----------|---------|
| 独立した知識として成立する | 専用のknowledge_nodeを作成する |
| あるノードに付随する補足情報 | そのノードの notes に記述 |
| ある英文に対する注釈 | その sentence の notes に記述 |
| 確認テストで問える知識 | check_points に追加 |

**判断基準**: 「この補足を知らないと、生徒が問題を間違える可能性があるか？」がYesなら、notesに埋もれさせず、check_pointsまたは独立ノードとして可視化する。

## 3. 3層アーキテクチャ

全参考書共通の構造化モデル。

```
Layer 1: 知識ノード（knowledge_nodes）
  → 参考書の各セクションから抽出した知識の単位
  → 理解目標（understanding_goals）と確認ポイント（check_points）を持つ
  → ノード間の前提関係（prerequisites）を定義

Layer 2: 英文データベース（sentences）
  → 参考書に掲載されている英文を登録
  → 構文分析（sentence_structures）を付与
  → 知識タグ（sentence_knowledge_tags）で「どのノードの実践か」を紐付け

Layer 3: セクションマッピング（sections）
  → 参考書のセクション構成を反映
  → セクション → ノードの対応関係（section_knowledge_nodes）を定義
  → セクション間の前提関係（section_prerequisites）を定義
```

## 4. セクション作業フロー（1セクションあたり）

### Step 1: 原本確認

1. 書き起こしMDを**全文**読む
2. PDF画像を開いて照合する
   - 誤字脱字、表の崩れ、記号の欠落を確認
   - 図解・樹形図・表・イラストの存在を記録
   - 太字・下線・色分けなどの強調表現を確認
3. 差異があれば書き起こしMDに注記する
4. **情報の棚卸し**: そのセクションに含まれる情報を列挙する
   - 概念の定義は何個あるか？
   - ポイント/ルールは何個あるか？
   - 例文は何個あるか？
   - 図解・補足・例外は何個あるか？
   - 難易度マーカーはあるか？
   - 他セクションへの参照はあるか？

> **目的**: 後のStep 2〜3で「情報の棚卸しリスト」と照合し、漏れを防止する。

### Step 2: 知識ノード抽出

1. **参考書別設計書を参照**し、そのセクションの構造に応じたノード抽出を行う
   - 参考書別設計書に記載された「ノード抽出ルール」に従う
   - 参考書の構造は書籍ごとに異なるため、共通手順を押し付けない
2. 各ノードに以下を定義する:
   - `id`: 参考書別設計書の ID 体系に従う
   - `name`: ノード名（簡潔・一意）
   - `category`: 参考書別設計書のカテゴリ体系に従う
   - `priority`: P1（基礎・必須）/ P2（標準）/ P3（発展・補足）
   - `notes`: ノードの詳細説明。図解情報・補足・例外もここに記述
3. 各ノードの `understanding_goals` を定義（「〜できる」形式、3〜6個）
4. 各ノードの `check_points` を定義（question/answer形式、3〜5個）
5. ノード間の `prerequisites` を定義
6. **棚卸しリストとの照合**: Step 1 で列挙した情報がすべてノードに反映されているか確認

### Step 3: 英文データベース作成

1. セクション内の英文を**すべて**登録する
   - 参考書別設計書の「英文抽出ルール」に従い、どの英文を登録対象とするか判断
   - 例文/練習問題/発展問題 etc. の role 分類は参考書別設計書に従う
2. 各英文に以下を定義する:
   - `id`: 参考書別設計書の ID 体系に従う
   - `english`: 英文（PDF画像と照合して正確に）
   - `japanese`: 和訳（PDF画像と照合して正確に）
   - `role`: 参考書別設計書で定義した分類
   - `notes`: 出典（大学名）、難易度マーカー等
3. 各英文の `sentence_structures` を定義
   - `overall`: 文型記号（S V O C M）
   - `details`: カタマリの範囲と役割を明示
4. 各英文の `knowledge_tags` を定義
   - その英文が**実際に練習・例示している**ノードのIDを付与
   - 1英文に複数タグ可（複合的な英文の場合）
5. **PDF画像での照合**: 英文のスペル、和訳の正確性を必ずPDF画像で確認

### Step 4: セクションマッピング

1. セクション情報を定義する:
   - `id`: 参考書別設計書の ID 体系に従う
   - `book`: 参考書名
   - `title`: セクションタイトル
   - `pages`: ページ範囲
   - `type`: introduction / drill / exercise / etc.（参考書別設計書に従う）
2. `section_knowledge_nodes`: このセクションで扱うノードを seq 付きで列挙
3. `section_prerequisites`: 前提セクションを定義
4. `knowledge_references`: ノード → 参考書内の位置（pages）を定義

### Step 5: 品質ゲート（5段階）

**全5ゲートをセクション単位で実施する。** 1つでも不合格ならStep 2〜4に戻って修正する。

#### Gate 1: 情報完全性チェック（漏れゼロ確認）

Step 1 の「情報の棚卸しリスト」と突合し、以下を確認:

- [ ] 参考書のセクションに書かれている**すべての概念・ルール**がノードとして登録されているか
- [ ] **すべての英文**が sentences に登録されているか
- [ ] 図解・樹形図の情報が notes に記述されているか
- [ ] 難易度・頻出度マーカーが保存されているか
- [ ] 書内の相互参照が保存されているか
- [ ] 補足・例外の情報が保存されているか

#### Gate 2: 自動バリデーション

`scripts/validate.py` を実行し、以下をチェック:

```
CHECK-01: YAML構文チェック（全ファイル）
CHECK-02: knowledge_nodes の必須フィールド
  → id, name, category, priority, understanding_goals, check_points, notes
CHECK-03: sentences の必須フィールド
  → id, section_id, drill, number, role, english, japanese, knowledge_tags
CHECK-04: knowledge_tags が参照するノードIDが実在するか
CHECK-05: knowledge_tags が空の英文がないか
CHECK-06: 英文が0件のノードがないか（警告レベル — introduction ノードは除外）
CHECK-07: section_knowledge_nodes のノードIDが実在するか
CHECK-08: prerequisites の循環依存がないか
CHECK-09: ID命名規則の一貫性（参考書別設計書の体系に準拠）
CHECK-10: manifest.yaml との整合性（セクション数・ノード数・英文数）
CHECK-11: understanding_goals の数（ノードあたり 3〜6 個）
CHECK-12: check_points の数（ノードあたり 3〜5 個）
CHECK-13: ノード/セクション比率（参考書別設計書の目安範囲内か）
```

#### Gate 3: 粒度・構造レビュー

別のClaudeエージェント（またはCodex）に、以下の観点でレビューを依頼する:

```
1. 粒度チェック:
   - 1つのノードに複数の独立した概念が混ざっていないか？
   - 逆に、細かすぎて統合すべきノードはないか？
   - 判断基準: 「このノードを 3〜5 個の check_points で完全にテストできるか？」

2. 依存関係チェック:
   - prerequisites は適切か？
   - 必要な前提が抜けていないか？
   - 不要な前提が設定されていないか？

3. 表記一貫性:
   - ID命名規則、カテゴリ名、フィールド名に揺れがないか？
   - notes の記述スタイルが統一されているか？
```

#### Gate 4: 原本忠実性レビュー

別のClaudeエージェント（またはCodex）に、**参考書の書き起こしMD（またはPDF）**と構造化データを渡し、以下を検証:

```
1. 情報ロスチェック:
   - 参考書に書かれている知識で、構造化データに反映されていないものはないか？
   - 補足・例外・注意書きが漏れていないか？

2. 情報歪みチェック:
   - 構造化の過程で、著者の意図と異なる解釈になっていないか？
   - ノードの notes は参考書の記述を正確に反映しているか？
   - understanding_goals は参考書の到達目標と合致しているか？

3. タグ妥当性:
   - 英文の knowledge_tags は適切か？
   - その英文が本当にそのノードの練習になっているか？
   - タグの過不足はないか？
```

#### Gate 5: 定量チェック

自動または手動で以下の指標を確認:

```
- ノード/セクション比率: 参考書別設計書の目安範囲内か（例: 1.5〜2.5）
- check_points/ノード: 3〜5 の範囲か
- understanding_goals/ノード: 3〜6 の範囲か
- 英文/セクション: 参考書別設計書の目安範囲内か
- knowledge_tags/英文: 1〜3 の範囲か（多すぎるとノード粒度が粗い可能性）
- 全英文のPDF照合が完了しているか
```

### Step 6: manifest 更新

manifest.yaml の該当セクションを更新:
- `status`: completed
- `knowledge_nodes_created`: ノード数
- `sentences_created`: 英文数
- `gates_passed`: [1, 2, 3, 4, 5]
- `reviewed_at`: 日時
- `notes`: 特記事項

### Step 7: コミット

```
feat(構造化): 〈参考書略称〉 〈セクションID〉 知識ノードN件・英文M件
```

## 5. データフォーマット仕様（YAML）

### 5.1 knowledge_nodes.yaml

```yaml
# 1参考書につき1ファイル、または章ごとに分割
# ファイル名: knowledge_nodes_〈参考書略称〉_〈章〉.yaml

knowledge_nodes:
  - id: "prefix-NNN"          # 参考書別設計書のID体系に従う
    name: "ノード名"            # 簡潔・一意
    category: "カテゴリ名"      # 参考書別設計書のカテゴリ体系に従う
    priority: "P1"             # P1（基礎・必須）/ P2（標準）/ P3（発展・補足）
    notes: |                   # 詳細説明（複数行可）
      ノードの詳細な説明。
      [図表: 樹形図 — 文型判別の手順（p.45）]
      参考書内参照: →p.174「後置修飾」
      例外: 〜の場合は適用されない

understanding_goals:
  - node_id: "prefix-NNN"
    seq: 1
    goal: "〜できる"            # 観察可能な行動で記述

check_points:
  - node_id: "prefix-NNN"
    seq: 1
    question: "〜とは何か？"
    answer: "〜である。"

node_prerequisites:
  - node_id: "prefix-NNN"
    prerequisite_id: "prefix-MMM"

knowledge_references:
  - node_id: "prefix-NNN"
    book: "参考書名"
    section_id: "セクションID"
    pages: "p.XX-YY"
```

### 5.2 sentences.yaml

```yaml
# 1参考書につき1ファイル、またはセクションごとに分割
# ファイル名: sentences_〈参考書略称〉_〈セクションID〉.yaml

sentences:
  - id: "sent-id"              # 参考書別設計書のID体系に従う
    section_id: "セクションID"
    drill: 1                   # ドリル/大問番号
    number: 1                  # 番号（ドリル内の連番）
    role: "example"            # 参考書別設計書で定義した分類
    english: "English sentence here."
    japanese: "和訳"
    notes: "出典大学名、難易度マーカー等"

sentence_structures:
  - sentence_id: "sent-id"
    label: "overall"           # overall: 文型全体 / details: 詳細分析
    value: "S V O"

sentence_knowledge_tags:
  - sentence_id: "sent-id"
    node_id: "prefix-NNN"
```

## 6. 作業順序（参考書ごと）

### Phase 0: 参考書別設計書の作成（必須・先行）

1. 書き起こしMD全体を**通読**する（最低でも各章の冒頭と代表セクション3つ）
2. 参考書の構造パターンを分析する
3. 「参考書別設計テンプレート」に沿って設計書を作成
4. **人間のレビューを受けて承認を得る**（承認前に着手しない）

### Phase 1: パイロット（最初の2〜3セクション）

1. 設計書に基づいて最初の2〜3セクションを構造化する
2. 5つの品質ゲートをすべて通す
3. **パイロットの振り返り**:
   - 設計書の粒度基準は適切だったか？
   - ノード抽出ルールは明確だったか？
   - 予想外の構造パターンはなかったか？
4. 必要に応じて設計書を修正する
5. 人間に報告し、方向性の確認を受ける

### Phase 2: 本格構造化

1. 残りのセクションを順次構造化する
2. 各セクションで5つの品質ゲートを通す
3. 章の区切りごとに進捗を報告する

### Phase 3: 全体整合性チェック

1. 全ノードの prerequisites を通しで確認（章をまたぐ依存関係）
2. 全ノードの粒度を通しで確認（章間で粒度の揺れがないか）
3. 定量指標の全体確認
4. 最終レビュー

## 7. コミット規約

## 9. 旧ワークフローからの変更点

> 本書は `docs/design/作業ワークフロー.md`（旧版）を置き換える。

| 旧版 | 本書 | 変更理由 |
|-----|------|---------|
| 「はじめ」固有の構造を前提 | 参考書非依存。固有部分は参考書別設計書へ | 複数参考書への対応 |
| 3段階レビュー | 5段階品質ゲート | 情報ロスと品質問題の検出精度向上 |
| Codexレビューはチャプター単位 | 全レビューをセクション単位に | 品質最優先の方針 |
| YAML構文チェック中心 | 情報完全性・原本忠実性チェックを追加 | 情報ロスゼロの実現 |
| manifest に pdf_verified / codex_reviewed | gates_passed リストに統合 | 5ゲート体制への対応 |
| Python モジュール形式（肘井） | YAML形式に統一 | レビュー容易性・LLM生成容易性 |
| 図解・マーカー等の扱い未定義 | §2 情報保存ルールで明確化 | 情報ロス防止 |
