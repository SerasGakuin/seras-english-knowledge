## 1. プロジェクトの背景

### 既存資産
- `seras-english-grammar` リポジトリに、参考書の書き起こしが進行中
- 核心・成川の2冊は書き起こし完了
- 他4冊（スクランブル、はじめの英文読解ドリル、肘井、入門英文問題精講）は目次のみ

### 塾の運営モデル
- 週1回・1時間の面談（生徒と大学生講師）
- 面談内容：1週間の勉強の仕方、質問対応、理解度チェック、来週の課題決定
- 基本は紙の参考書で自習させるスタイル

### 技術環境
- 生徒：紙ベース（参考書、ノート、確認テスト）
- 講師：Googleエコシステム（Docs、Sheets、GAS、Web app）
- 進捗管理はGoogle Sheetsで一元管理、APIで連携

## 2. 解決したい課題

### 最上位目標
**生徒の定着度を上げる**

### 「定着していない」の内訳

#### A. 理解の問題
- 読んだつもりで本質を掴めていない
- どこを理解すべきかわからない
- 「問題が解ければOK」という意識で、理論を理解しようという動機がない

#### B. 記憶の問題
- 理解したけど忘れる
- 復習のタイミングが体系化されていない

#### C. 応用の問題（文法→読解の接続不全）
- 文法知識と文構造意識（SVOCM、句・節）が繋がっていない
- 例：関係代名詞は知っているが、それが「形容詞節を作る」と知らない
- 例：分詞構文は知っているが、それが「分詞が副詞として働いている」と理解していない

### 構造的な問題

#### 参考書間の構成軸の違い
| 文法参考書（核心、成川、スクランブル） | 読解参考書（はじめの〜、肘井、入門精講） |
|---|---|
| 文法項目で章立て | 読解の視点で章立て |
| 仮定法、関係詞、不定詞... | SVの発見、カタマリ、識別、倒置... |

→ 生徒は自力で知識を接続できない

#### 講師の能力差
- 「理論が大事」という意識はある
- しかし限られた面談時間で「具体的にどこを、どう読め」まで落とし込めない
- 結果「問題が解けるだけじゃダメだよ〜」と一言で終わる

## 3. システムの方向性

### 基本思想
- 参考書学習が主、システムは補助
- 人の意識を変えるのは人、システムはそれを補助する
- 講師の能力に依存せず、枠組みでアシストする

### システムが担う役割

#### 「具体化」の代行
- 「この章を理解した」とは具体的に何ができる状態かを言語化
- 講師が説明しなくても、生徒が確認できる
- 講師も「これ説明できる？」と聞くだけで確認できる

#### 参考書の「標準化」
- 参考書に依存しない「あるべき知識体系」を定義
- 各参考書をそこにマッピング
- 抜け漏れや癖を補完する相互参照を提供

#### 文法と読解の「橋渡し」
- 文法用語と文構造用語の対応を明示
- 「関係代名詞 = 形容詞節を作る」を明確に伝える
- 文法→読解、読解→文法の双方向マッピング

## 4. 標準化された知識体系のイメージ

### 知識ノードの構造（例：関係代名詞）

```
■ 関係代名詞

【文法としての理解】
  → 核心 Part4_21 / 成川 Part07

【文構造での役割】
  → 形容詞節を作る
  → 修飾する名詞（先行詞）の直後に置かれる

【読解での応用】
  → 肘井 テーマ05「形容詞節で英文が読める」
  → はじめの〜 Chapter4-2「形容詞節① 関係代名詞」

【チェックポイント】
  □ 関係代名詞が作る節は何節か言える
  □ 先行詞を指して「これを修飾している」と説明できる
  □ who/which/thatの使い分けを説明できる
```

### 文法用語と文構造用語の対応表

```
【準動詞・関係詞 → 文中での役割】

関係代名詞 ──→ 形容詞節を作る（名詞を修飾）

分詞 ─┬─→ 形容詞として働く → 名詞を修飾
      └─→ 副詞として働く → 分詞構文

動名詞 ──→ 名詞として働く → S/O/Cになる

不定詞 ─┬─→ 名詞的用法 → S/O/Cになる
        ├─→ 形容詞的用法 → 名詞を修飾
        └─→ 副詞的用法 → 動詞/形容詞/文を修飾
```

## 10. 再現性・柔軟性への対応方針

### 設計思想
- 知識ノードは**参考書に依存しない「標準体系」**として設計
- 参考書へのマッピングは別レイヤー（追加・削除が容易）
- フォーマットを明確に定義し、**誰でも同じ形式で追記できる**ようにする

### 集合知システム（Phase分け）

**Phase 1（必須・2ヶ月以内）**
- 知識ノードのフォーマット確定
- Google Formで簡易フィードバック収集
- フィードバック→修正のプロセスを文書化

**Phase 2（できれば）**
- 講師向けWebアプリで知識ノード閲覧・コメント
- コメントを元に修正する仕組み

**Phase 3（将来）**
- テスト結果との連携
- AIによる改善提案

## 11. システムの対象ターゲット

### ターゲット
- **理論派**（理論的な文法理解ができる/目指す生徒）
- **関関同立以上、国公立志望**

### 対象外
- **感覚派** → 別アプローチ（多読・音読中心）
- 理論的な文法理解が合わない生徒

### 背景
- SVOCMや「名詞節」「形容詞句」といった用語を使った理解は、高校生には難しい場合がある
- 理論が苦手な生徒に理論を強制しても、お互い不幸になる
- 入塾時から「理論派」「感覚派」で分けるべきでは、という校舎長との議論あり

## 12. 知識ノードのフォーマット設計

### ファイル形式
**YAML**を採用（人間が読みやすい、コメント可能）

### フィールド定義

```yaml
# === 必須フィールド ===
id: rel-001                    # 一意のID（3文字プレフィックス + 3桁番号）
name: 関係詞の基本的な働き       # 知識ノード名
category: 関係詞/総論           # カテゴリ（階層）
priority: P1                   # 優先度（P1/P2/P3）

understanding_goals:           # 理解すべきこと（箇条書き）
  - 関係詞は前の名詞に情報を追加する
  - 関係詞が作るカタマリが何節かを意識する

check_points:                  # 確認ポイント（これが言えればOK）
  - 関係代名詞が作る節は何節か言える
  - 関係副詞が作る節は何節か言える

references:                    # 参考書マッピング
  核心:
    pages: p.300-301
  成川:
    pages: p.286-288

# === オプションフィールド ===
structure_role:                # 文構造での役割（読解との接続）
  - 関係代名詞 → 形容詞節を作る
  - 関係代名詞what → 名詞節を作る

prerequisites:                 # 前提知識（他ノードのID）
  - gram-015

related:                       # 関連知識
  - rel-002
  - rel-010

notes:                         # 補足メモ（講師向け）
  - 生徒は「形容詞節」という言葉自体を知らないことが多い

examples:                      # 例文（あれば）
  - sentence: "The book which I bought is interesting."
    explanation: whichは形容詞節を作り、The bookを修飾
```

### 優先度の定義

| 優先度 | 意味 | 対象 |
|---|---|---|
| P1 | 基礎理論 | 理論派なら全員必須。これがないとP2以降が積み上がらない |
| P2 | 標準理論 | 関関同立・中堅国公立で必要。入試頻出 |
| P3 | 発展理論 | 難関国公立・早慶向け。細かい識別、例外、難問対策 |

### ID体系

```
rel-xxx  → 関係詞
hyp-xxx  → 仮定法
inf-xxx  → 不定詞
prt-xxx  → 分詞
ger-xxx  → 動名詞
...
```

### カテゴリ例（関係詞）

```
関係詞/総論
関係詞/関係代名詞
関係詞/関係副詞
関係詞/what
関係詞/複合関係詞
関係詞/読解接続
```

## 14. 確認テストのデータ構造

### 基本方針（決定済み）
- **紙ベース**が基本（面談時に使用）
- 形式は問題に合わせて柔軟に組み合わせる（一問一答/説明型/選択式）
- **check_points = ゴール定義**（知識ノード内）、**確認テスト = それを確認する手段**（別管理）
- check_pointsと似た内容を確認する問題を出す
- 他の参考書から同範囲の少し難しい類題を出す

### 問題の2つの種類
1. **オリジナル確認問題** - check_pointsに対応する問い（「関係代名詞が作る節は何節？」など）
2. **参考書の類題参照** - 「スクランブル p.XXX Q3を解かせる」のように参照だけ記載（著作権配慮で問題文は書かない）

### 構造案

```yaml
# tests/rel-001.yaml（知識ノードと1対1）

target_node: rel-001

# ① オリジナル確認問題
questions:
  - id: rel-001-q1
    type: short_answer        # short_answer / explanation / choice
    question: "関係代名詞が作る節は何節ですか？"
    answer: "形容詞節"

  - id: rel-001-q2
    type: explanation
    question: "関係代名詞と関係副詞の後ろの文の違いを説明してください"
    answer_points:             # 説明型はポイント列挙
      - 関係代名詞の後ろは不完全な文
      - 関係副詞の後ろは完全な文

  - id: rel-001-q3
    type: choice
    question: "次のうち形容詞節を作るものはどれか"
    options: [関係代名詞, 接続詞that, 関係代名詞what]
    answer: 関係代名詞

# ② 参考書の類題参照
related_exercises:
  - book: スクランブル
    pages: p.200-201
    problems: "Q1〜Q5"
    note: "基本的な識別問題"
  - book: 英文法問題Solution
    pages: p.150
    problems: "1-3"
    note: "少し応用的"
```

### 未決定事項

#### A. ファイルの粒度
- 知識ノードと1対1（`tests/rel-001.yaml`） → シンプルだがファイル数が多い（400〜500）
- カテゴリ単位でまとめる（`tests/関係詞.yaml`） → ファイル数は減るが1ファイルが長い

#### B. typeの種類
- `short_answer`（一問一答）、`explanation`（説明型）、`choice`（選択式）の3つで足りるか？
- 他に必要な形式はないか？

#### C. related_exercisesの粒度
- 問題番号まで指定する？「この辺のページ」くらいでいい？
